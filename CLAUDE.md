# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

This is a **cookiecutter template**, not a Python package. It generates a scaffold for a Python 3 package. There are two distinct layers here, and it is important not to confuse them:

- **The template repo itself** (this top level): `cookiecutter.json`, `hooks/`, `tests/`, root `Makefile`. Editing these changes *the template*.
- **The template payload** in `{{cookiecutter.project_slug}}/`: every file under this directory is copied (with Jinja2 rendering) into a generated project. The `Makefile`, `pyproject.toml`, etc. *inside* this folder belong to the *generated* project, not to this repo.

## Commands

Run from the repo root:

- `make test` — bootstraps a `.venv` (installs pytest-cookies plus pinned black and isort) and runs the template test suite via `pytest --ignore=tests/examples`. Test options are `-v -x -s`.
- `make clean` — removes the `.venv`.
- Run a single test config: `.venv/bin/pytest tests/test_template.py -k minimal` (config names: `default`, `minimal`, `only_latest`, `rst_docs`, `custom_metadata`).

The test suite is slow: each configuration bakes a real project and runs `make test`, `make docs`, `make dist`, git init/commit, etc. inside it via subprocess. `uv`, `make`, and `git` must be on PATH; `uv` downloads the generated project's interpreters (3.10–3.14) on demand.

## How the template is rendered

- **Variables** live in `cookiecutter.json`. `project_slug` is derived from `project_name`; the `requires-python` floor and the lowest interpreter used by `test-lowest` derive from the lowest enabled `support_pyXX` flag.
- **Conditional content** is Jinja2 *inside file bodies* (not just filenames). See `{{cookiecutter.project_slug}}/pyproject.toml` and `Makefile` for heavy use of `{% if cookiecutter.sphinx_docs == 'y' %}` etc. When editing these files, remember the output must remain valid for *all* combinations of flags — black/isort/flake8 are run against generated output in tests, so whitespace and formatting in the rendered result matter.
- **`hooks/pre_gen_project.py`** validates the slug and warns about missing executables before generation.
- **`hooks/post_gen_project.py`** deletes files for disabled features (e.g. removes `docs/` when `sphinx_docs == 'n'`, `scripts/release.py` when `on_pypi == 'n'`) and optionally runs interactive git setup.

## How tests work

`tests/test_template.py` is the whole suite. `CONFIGURATIONS` is a list of `(name, context, checkers)` tuples; `test_template` is parametrized over it. Each `context` overrides `cookiecutter.json` defaults, and each `checker` is a function receiving the baked project path.

`tests/examples/<name>/` holds **golden reference outputs** for the `default` and `minimal` configs. `check_default_files`/`check_minimal_files` compare the generated file list against these. If a referenced example folder is missing it is regenerated by baking the template — so **deleting an example and re-running tests recreates it, and you must verify the new contents by hand** before trusting them. Changing which files the template emits requires updating these reference trees.

When you change `make help` targets in the template, update the expected output in `check_default_make_help` / `check_minimal_make_help`, which assert against the exact help text.

## Generated-project model (for context when editing the payload)

The generated project uses a `src/` layout and a `uv`-based workflow: a `Makefile` whose targets call `uv run`/`uv build`/`uv sync` against PEP 735 dependency groups defined in `pyproject.toml` (the single source of metadata and tool config). `PYTHON` and `RESOLUTION` parameterize the Makefile; `make test-lowest` exercises the lowest supported interpreter with `--resolution lowest-direct`. `pre-commit` enforces black + isort. Docs are Sphinx, deployed to GitHub Pages with a [Docs Versions Menu](https://github.com/goerz/docs_versions_menu) (not ReadTheDocs); the `docs-versions-menu` Sphinx extension injects the menu at build time and the `docs-versions-menu` CLI regenerates `versions.json` on the `gh-pages` branch in the docs workflow. `scripts/` contains `release.py` (used by `make release`; removed when `on_pypi == 'n'`) and `check_changelog.py` (stdlib-only, always present; backs `make check-changelog` — textual validation of `CHANGELOG.md`, also run via `make lint` and CI — and `make changelog`, whose `--fix` mode queries the GitHub API to add missing `[#N]` link definitions).

The generated project tracks user-facing changes in a root `CHANGELOG.md` in [Keep a Changelog](https://keepachangelog.com) format, included into the docs via `docs/sources/changelog.md`/`.rst`. The fresh file has only an `## [Unreleased]` section (no release headings yet); `check_changelog.py` deliberately tolerates that pre-first-release state. `make release` (`scripts/release.py`) transforms the changelog mechanically: it renames `## [Unreleased]` to `## [vX.Y.Z] - date` and rewrites the version links for the release commit, builds the signed tag message and GitHub release (via `gh`) from that section's notes, and re-adds a fresh `## [Unreleased]` heading in the `+dev` version-bump commit.

The `sphinx_docs` flag toggles docs on/off. When docs are enabled, two independent flags select the format:

- `markdown_docs` (default `y`): doc *source files* (index, readme, etc.) are MyST markdown. The API page is generated by `sphinx-autodoc2` (statically, respecting each module's `__all__`) into a git-ignored `docs/sources/apidocs/` tree pulled in via a `:glob:` toctree in `index.md`. When `n`, the legacy `.rst` sources plus `sphinx.ext.autodoc`/`autosummary` are used, and the API page is the hand-written `api.rst` (a single `automodule` with `:members:`). `post_gen_project.py` deletes whichever source set is unused (`api.rst` only exists for the RST path).
- `markdown_docstrings` (default `n`, only meaningful when `markdown_docs == 'y'`): when `y`, autodoc2 parses docstrings as MyST markdown. When `n` (default), autodoc2 treats docstrings as plain RST (`:param:`, `:returns:` field lists). Note that `sphinx.ext.napoleon` only hooks into `sphinx.ext.autodoc` events, not autodoc2, so Google/NumPy-style napoleon syntax is not available in the `markdown_docs == 'y'` path regardless of `markdown_docstrings`.

Because both API generators respect `__all__`, every module must declare one (the template's `__init__.py` already mandates the `__all__`/`__private__` convention).

When `sphinx_docs == 'y'`, the generated package's `__init__.py` ships an example `hello_world` function (listed in `__all__`) so the API page is non-empty out of the box. The docstring style matches the settings: MyST when `markdown_docstrings == 'y'`, plain RST field lists when `markdown_docs == 'y'` and `markdown_docstrings == 'n'`, and Napoleon Google-style RST when `markdown_docs == 'n'`. Its doctest runs under `make test` (via `--doctest-modules`), there is a `test_hello_world` unit test, and the `check_api_docs` checker in `tests/test_template.py` asserts the built HTML contains the function.
